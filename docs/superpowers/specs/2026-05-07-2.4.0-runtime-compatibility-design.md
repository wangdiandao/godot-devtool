# 2.4.0 Runtime Compatibility Design

## Goal

Complete the `2.4.0 Runtime Compatibility Completion` roadmap by making runtime input and recording tools executable through the installed Godot runtime bridge, updating capability descriptions so clients can distinguish real runtime routes from compatibility wrappers, and synchronizing release metadata and bilingual docs.

## Architecture

Runtime behavior remains owned by the Godot addon command router. `src/addons/godot_devtool/commands/runtime_commands.gd` is the source addon implementation copied into builds, while `src/godot/editorBridge.ts` generates the installed runtime bridge used by `plugin_install`. Both surfaces must expose the same runtime input and recording semantics.

The TypeScript MCP layer continues routing exact-name compatibility tools through `queueRuntimeCompatibilityCommand`. Capability metadata is improved at the tool-definition layer so `get_capabilities` and README generation describe whether a compatibility entry is a direct alias, a native/bridge-backed exact-name wrapper, or a canonical-route wrapper.

## Runtime Input Requirements

- `simulate_key` creates `InputEventKey`, accepts numeric keycodes or Godot key names, validates that the key resolves, applies `pressed`, and injects with `Input.parse_input_event`.
- `simulate_mouse_click` creates `InputEventMouseButton`, accepts `button`, `buttonIndex`, `pressed`, and position payloads, then injects through `Input.parse_input_event`.
- `simulate_mouse_move` creates `InputEventMouseMotion`, accepts position and relative vectors, then injects through `Input.parse_input_event`.
- `simulate_action` validates that the requested action exists in `InputMap`, supports press and release, clamps strength between `0.0` and `1.0`, and reports the applied strength.
- `simulate_sequence` accepts `events` or `sequence`, rejects non-array input, preserves per-event results, honors `delayFrames`/`delay_frames`, and returns `ok: false` when any event fails.

## Recording Requirements

- `start_recording` and `stop_recording` are registered runtime routes in both source addon and installed bridge.
- `_input(event)` captures key, mouse button, and mouse motion events while recording is active.
- `stop_recording` writes JSON to a `res://` path, creates parent directories, and returns clear file-write errors.
- `replay_recording` reads JSON, validates the events array, replays each captured event through the same input simulation path, honors frame pacing, and returns clear read/parse errors.

## Metadata And Docs

- Compatibility tool descriptions must not imply every exact-name route is only a completion receipt.
- Runtime input/recording tools should be described as runtime bridge routes in generated README tables.
- The packaged `godot-devtool` Skill states that it is a workflow guide, not the exhaustive tool catalog.
- `package.json`, `package-lock.json`, Skill metadata, README links, changelogs, and roadmaps are synchronized to `2.4.0`.

## Verification

- Static verification scripts check that source addon runtime commands and generated installed bridge both contain input injection and recording logic.
- Tool-definition verification checks runtime compatibility metadata and rejects weak placeholder markers.
- Roadmap verification checks version/docs synchronization and Skill coverage.
- Build output is regenerated with `npm run build`, then `npm run verify:tools`, `npm run verify:plugin`, `npm run verify:runtime`, and `npm run verify:roadmap` are run fresh.
