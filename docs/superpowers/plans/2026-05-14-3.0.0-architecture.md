# godot-devtool 3.0.0 Architecture Development Plan

Status: completed locally, not pushed or published.

## Scope

3.0.0 is the architecture release for shared MCP usage, multiple AI agents, multiple running Godot game instances, lower context usage, and predictable bridge-port lifetime.

The implementation keeps the existing public capability surface and adds management tools for the new architecture. It does not remove existing editor, runtime, native, headless, browser visualizer, compatibility, or project workflow tools.

## Architecture

```text
MCP clients / AI agents
  -> stdio MCP server instances
  -> shared WebSocket broker on the configured port
  -> editor bridge sessions identified by projectPath + sessionId
  -> runtime bridge sessions identified by projectPath + runId/sessionId
  -> run registry for multiple game/editor processes
```

Key boundaries:

- `src/server/broker/*`: broker client registry, command routing, leases, and status snapshots.
- `src/server/targets/*`: live bridge target resolution and ambiguity payloads.
- `src/server/runs/*`: multi-run process registry, output buffers, selection, status, and process lifecycle.
- `src/server/transports/wsBridge.ts`: shared broker transport, remote forwarding, reconnect waiting, and cleanup.
- `src/tools/definitions/*` plus `src/server/routeRegistry.ts`: compact route metadata and workflow-scoped capability discovery.
- `src/addons/godot_devtool/editor/*` and `src/addons/godot_devtool/runtime/*`: split Godot addon implementation with thin entry scripts.

## Sub-Agent Distribution

- Main thread: integration contracts, `GodotServer` method wiring, duplicate `runId` protection, cleanup behavior, version/docs/verifier sync, build/test/Godot validation.
- Worker A: `src/server/broker/*`, `src/server/targets/*`, and `src/server/transports/wsBridge.ts`; shared broker, leases, target ambiguity, and remote forwarding.
- Worker B: `src/server/runs/*`; multi-instance run registry, selection, status, output read/clear, and process lifecycle helpers.
- Worker C: `src/addons/godot_devtool/**`; thin plugin/autoload entries, editor bridge client, status dock, runtime client, and runtime state store.
- Worker D: `src/tools/**`, `src/server/handlers/core.ts`, and `src/server/routeRegistry.ts`; 3.0 tool definitions, workflow filters, handler routing, and route metadata.
- Worker E capacity was not opened; documentation, verifier integration, and final verification were kept on the main thread to avoid write-scope overlap.

## Completed Capabilities

- Shared MCP broker lets multiple MCP clients or AI agents use one WebSocket bridge port.
- Multiple game instances are supported through `runId`; the registry supports caller-provided IDs, rejects duplicates before spawning, and lets debug/stop tools target a specific run.
- `list_bridge_sessions`, `resolve_bridge_target`, and structured ambiguity errors prevent silent selection when multiple editor/runtime sessions match.
- `broker_status`, `broker_cleanup_idle`, and broker leases expose current clients, pending commands, and safe idle cleanup.
- `get_capabilities` workflow filters keep context small while preserving the full tool catalog.
- Runtime listener lifecycle keeps the bridge alive while a run or runtime client needs it and releases the port once no live state remains.
- Godot addon files are split at design time into small editor/runtime modules rather than collecting new logic in one large script.
